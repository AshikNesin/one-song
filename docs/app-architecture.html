One Song

A minimal Android music player that plays a single song on repeat — built with React Native. This doc covers the full architecture, every component, service, and lesson learned.

01 What is One Song?

One Song is an Android-only music player that does exactly one thing: play a single audio file on infinite loop. No playlists, no libraries, no complexity. Pick one song from your device and the app remembers it forever.

Core Features

Instant Auto-PlaySong starts immediately on launch — no taps needed
Infinite LoopUses native RepeatMode.Track for gapless looping
Sleep Timer5 to 60 minutes, configurable default in Settings
Background AudioPlays while locked, in other apps, even when app is killed
Notification ControlsPlay/Pause from lock screen and notification shade
Seekable ProgressTap or drag the progress bar to jump anywhere
Persistent SongRemembers your song even if you move or rename the original file
In-App UpdatesDetects and installs Play Store updates without leaving the app

Why So Simple?

The app is intentionally single-purpose. Rather than building a general music player, One Song solves a specific problem: you have one song you want to listen to, always — a meditation track, a white noise loop, a focus beat. Every design decision flows from this constraint.

Design Decisions

No Global State Library

No Redux, no Context, no Zustand. A simple module-level pub/sub pattern in Playback.ts is sufficient for a single-screen player app. State lives outside React, enabling the 1-second polling loop and audio focus handlers to work independently of component lifecycle.

1-Second Polling for Position

Instead of event-driven position tracking (complex with native audio), a simple 1-second setInterval reads position/duration from the native player. Trade-off: ~1% CPU overhead vs much simpler code.

Song Copied to App Cache

Selected files are copied to cachesDirectory to avoid Android content URI permission expiration on app reinstall. Trade-off: files may be cleared by OS under storage pressure, but simpler than documentsDirectory.

03 Audio Playback System

The audio engine is react-native-track-player, which wraps Android's native MediaSession system. Audio runs in a foreground service, meaning playback continues even when the app is backgrounded or killed.

Audio runs in a foreground service — survives app kill. Notification auto-generated by TrackPlayer.

Track Lifecycle

StepAPI CallWhat Happens
1. Setup TrackPlayer.setupPlayer() Initializes the native audio engine. Configures capabilities (Play, Pause, SeekTo), sets RepeatMode.Track for infinite loop, and sets StopPlaybackAndRemoveNotification for app-kill behavior
2. Load TrackPlayer.reset() + add() Clears any existing track queue, adds a single track with { id, url, title, artist, artwork, duration }
3. Play TrackPlayer.play() Starts playback. Android creates a media notification automatically showing track info and Play/Pause controls
4. Poll getProgress() + getPlaybackState() Every 1 second, reads current position/duration and play state from the native player, updates React state
5. Loop RepeatMode.Track Native looping at the audio engine level — no JS intervention needed, gapless

Audio Focus Handling

What happens when you get a phone call?

TrackPlayer integrates with Android's audio focus system. When another app requests focus:

  • Transient focus loss (notification sound, alarm) → auto-pause, auto-resume when focus regained
  • Permanent focus loss (phone call, other music app) → pause and stay paused
  • Focus regained → resume playback automatically (transient only)

Handled via useTrackPlayerEvents([Event.RemoteDuck]) → maps to AudioFocusEvent objects → updates module-level state.

Remote Controls (Notification)

The notification shade and lock screen show:

  • Album artwork (from ID3/MP4 metadata)
  • Track title & artist
  • Play/Pause button

These are auto-managed by TrackPlayer. Tapping a notification button triggers Event.RemotePlay / Event.RemotePause which our useRemotePlayPause() hook handles.

Android Native Configuration

<!-- AndroidManifest.xml -->
<service
  android:name="com.doublesymmetry.trackplayer.service.MusicService"

The MEDIA_BUTTON intent filter allows headset buttons to control playback. The JS registration is required by the library even if the callback is empty — all event handling happens via hooks.

04 Data & State Management

One Song uses a module-level pub/sub pattern instead of Redux, Context, or Zustand. This is a deliberate choice — the app is a single-screen player with simple state.

The Pub/Sub Pattern

State Flow
Playback.ts
→
State Object + Listeners Set
→
PlaybackController.ts
→
PlayerScreen.tsx
1. State changes → notify() → all subscribe()d React components re-render 
// Playback.ts — simplified
let state: PlaybackState = { isPlaying: false, position: 0, /* ... */ };
const listeners = new Set<(s: PlaybackState) => void>();

export function subscribe(cb) { listeners.add(cb); return () => listeners.delete(cb); }
export function getState() { return state; }

function notify() { listeners.forEach(cb => cb(state)); }

// Usage in React hook:
export function usePlaybackController() {
  const [s, setS] = useState(getState());
  useEffect(() => subscribe(setS), []);  // React re-renders on every notify()
}

Dual Metadata Extraction: ID3 + MP4

id3-parser only handles MP3 ID3 tags. M4A/MP4 files store metadata in MP4 atoms — a completely different binary structure. The custom parseMp4Metadata() walks the atom tree to find moov/udta/meta/ilst, then reads child atoms for title (©nam), artist (©ART), and cover art (covr).

06 Service Layer

AudioService.ts

Thin wrapper around react-native-track-player. Provides setupPlayer(), loadSong(), play(), pause(), seekTo(), getProgress(), getPlaybackState(). Also exposes React hooks: useRemotePlayPause() for notification controls and useAudioFocus() for inter-app audio handling.

Playback.ts + PlaybackController.ts

Playback.ts is the core state machine — NOT a React hook. Module-level state + listeners. Orchestrates init(), togglePlay(), seek(), startPolling(), and handleAudioFocus().

PlaybackController.ts is the React bridge — usePlaybackController() hook that subscribes to Playback state changes, runs init on mount, starts/stops polling, and registers remote/audio focus handlers.

SongIntake.ts

The file-picking pipeline: intake() orchestrates permission → file pick → copy to cache → metadata extraction → returns Song. complete() persists to AsyncStorage. getSong(), hasCompletedOnboarding(), clearSongData() manage the onboarding gate.

SleepTimer.ts

Simple setTimeout-based timer. setTimer(minutes, onExpire) sets a timeout. restoreTimer() re-arms on app launch so it survives restarts. saveDefaultTimer() / loadDefaultTimer() persist the default preference to AsyncStorage.

StorageService.ts

Typed AsyncStorage wrapper: getItem(key), setItem(key, value), removeItem(key), multiRemove(keys). All keys prefixed with @onesong:.

PermissionService.ts

Wraps react-native-permissions. Requests READ_MEDIA_AUDIO (Android 13+) or READ_EXTERNAL_STORAGE (Android 12-). Detects permanently denied state and offers an "Open Settings" path.

InAppUpdateService.ts

Wraps sp-react-native-in-app-updates (patched). Only active in non-dev builds. Checks for updates on app launch, handles FLEXIBLE (background download + install) and IMMEDIATE (blocking) update types. Multiple bugs fixed via patch-package (see TIL section).

07 React Native Basics (for newcomers)

What is React Native?

React Native lets you write mobile apps using React-style JavaScript/TypeScript that renders to real native UI components, not web views. A <Text> becomes an Android TextView. A <View> becomes an Android ViewGroup. This gives you native performance with a React development experience.

Key Concepts Used in One Song

JS Thread vs Native Thread

React Native runs JavaScript on a separate thread from the native UI. This is why audio can keep playing even when the JS thread is frozen — the native audio service runs independently. TrackPlayer operates entirely on the native side, with JS only sending commands and receiving status updates.

Native Modules

Libraries like react-native-track-player and react-native-device-info have both JavaScript code (the API you call) and native Java/Kotlin code (the actual implementation). React Native "autolinks" these at build time. If ProGuard/R8 obfuscates the Java classes, the JS side gets null back — a common source of release-build-only crashes.

Hermes JS Engine

React Native 0.85 defaults to Hermes, a lightweight JavaScript engine optimized for mobile. Unlike browser engines (V8/JSC), Hermes doesn't define window as a global. Accessing window.document throws ReferenceError instead of returning undefined — use typeof window === 'undefined' guards.

Path Aliases (@/)

One Song uses @/ as an alias for src/. This requires three separate configs:

  • tsconfig.json — "paths": {"@/*": ["./src/*"]} for type-checking
  • metro.config.js — custom resolveRequest for bundling at runtime
  • jest.config.js — moduleNameMapper for test resolution
AsyncStorage

React Native's equivalent of localStorage. It's a simple key-value store backed by native SQLite/SharedPreferences. All values are strings — you JSON.stringify objects before storing and JSON.parse when reading. One Song uses it for persisting the selected song, onboarding state, sleep timer default, and auto-play preference.

AndroidManifest.xml

The central configuration file for Android apps. Declares app name, icon, permissions (READ_MEDIA_AUDIO, POST_NOTIFICATIONS), services (MusicService for audio), and the launcher activity. Every native library that needs background services declares its hooks here.

ProGuard / R8 (Code Obfuscation)

Release builds run R8, which removes unused code and renames Java classes to short names. This breaks React Native's autolinking for native modules unless you add keep rules in proguard-rules.pro. Every native module library needs a keep rule. The TIL section has detailed examples of crashes caused by missing rules.

ABI Splitting

Android devices use different CPU architectures (arm64-v8a, x86, etc.). Native libraries (.so files) must match the architecture. Building for all architectures creates a 56MB APK — but the device only needs one. ABI splitting builds separate APKs per architecture, reducing size to ~14MB for modern phones.

versionCode vs versionName

versionCode (integer) is Google Play's internal identifier — must be strictly increasing for every upload. versionName (string like "1.0.1") is what users see. Both must be bumped together for every release.

08 Key Lessons (from TIL.md)

Every bug, fix, and architectural insight from building One Song. The full log is in TIL.md — these are the most impactful entries.

Bug Android 12+ Double Splash Screen

Android 12+ mandates a system splash screen that shows the app icon before onCreate() runs. Having both a system splash and a custom SplashActivity caused a double flash. Fix: removed SplashActivity, made MainActivity the launcher, customized the system splash via values-v31/styles.xml. The system splash cannot render text — text must be shown in the RN app.

Bug ProGuard/R8 Crashes on Release Builds

R8 obfuscates Java classes in release builds, renaming native module package classes and stripping @ReactModule annotations. This breaks React Native's autolinking. Fix: add -keep rules in proguard-rules.pro for every native module library. Debug builds don't run ProGuard, so these crashes only appear in production.

Fix In-App Update Stuck in DEVELOPER_TRIGGERED

The sp-react-native-in-app-updates library had two bugs: checkNeedsUpdate() returned shouldUpdate: false for DEVELOPER_TRIGGERED state, and startUpdate() rejected it. Patched via pnpm patch at both JS and Java levels. Lesson: DEVELOPER_TRIGGERED is a legitimate update state, not an error.

Fix Flexible Update Downloaded But Never Installed

You must call installUpdate() after the download completes. Old code was fire-and-forget — no status listener registered. Fix: register addStatusUpdateListener before startUpdate(), call installUpdate() on DOWNLOADED status. The listener must come first — status callbacks can fire synchronously for already-completed states.

Fix Release APK Signed with Debug Certificate

React Native's starter template defaults release builds to signingConfigs.debug. Must change to signingConfigs.release with env-var-based credential injection. Always validate credentials before the Gradle build — silent failures produce invalid signatures.

Arch Metadata Extraction: ID3 + MP4 Dual Path

id3-parser only handles MP3. M4A/MP4 files store metadata in a completely different binary format (MP4 atoms). Custom parser walks moov→udta→meta→ilst to find ©nam, ©ART, covr atoms. Key gotcha: meta is a "full box" with a 4-byte header before children; udta is not. MP4 data atom type indicator is the first byte, not a uint32. Artwork saved as file:// URI (not base64).

Arch Hermes: No window Global

Hermes doesn't define window. Accessing an undeclared variable throws ReferenceError (not undefined like in JSC/browsers). RN 0.85's DevTools setup code hit this. Fix: typeof window === 'undefined' guard, persisted via pnpm patch.

Fix APK Size: 56MB → 14MB

Building for all 4 CPU architectures packs 4 copies of every native library. ABI splitting targeting only arm64-v8a reduces APK size by 75%. Modern phones (2015+) all use arm64. Google Play's AAB format does this per-device automatically.

Bug versionCode Must Be Unique Per Upload

Google Play rejects uploads with duplicate versionCode. Only versionName was bumped; versionCode stayed at 1. Both must be bumped: versionCode for Play Store uniqueness, versionName for users. Automated via scripts/bump-android-version.sh.

Arch Java 17 Required for RN 0.85

React Native's New Architecture (TurboModules/Fabric) + NDK/CMake is sensitive to Java version. Java 25 (latest) throws restricted method errors. Pin Gradle to OpenJDK 17 via org.gradle.java.home in gradle.properties and export JAVA_HOME for the CLI. System default Java can stay at 25 — only the Android build needs 17.
